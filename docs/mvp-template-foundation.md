# Merchshop MVP Template Foundation

## Назначение документа

Документ фиксирует первый продуктовый контур `Merchshop` как универсального шаблона, который может внедряться внутри разных компаний для программ нематериальной мотивации сотрудников.

Фокус текущего этапа:

- только `MVP`
- продуктовый шаблон, а не кастом под одну компанию
- без блока `KPI`
- без клиентских интеграций в первой версии

## Базовые допущения

1. `Merchshop` разрабатывается как повторно используемое продуктовое ядро.
2. Один инстанс `MVP` обслуживает одну компанию.
3. `MVP` поставляется как web-приложение.
4. Базовый язык интерфейса в первой версии: русский.
5. Первая версия должна работать без `SSO`, `HRIS`, `ERP`, `WMS` и `BI`.
6. Различия между компаниями должны по максимуму закрываться настройками, а не переписыванием логики.

## Границы MVP

### Что входит в MVP

- локальная авторизация без внешних провайдеров
- две базовые роли: `Employee` и `Program Admin`
- импорт сотрудников из Excel
- справочник сотрудников со статусами
- журнал транзакций для всех операций, влияющих на баланс
- снимок текущего баланса для быстрых чтений
- ручные начисления администратором
- ручные корректировки баланса с обязательной причиной и комментарием
- единое глобальное правило сгорания баллов
- каталог товаров с ручным управлением остатками
- оформление заказа из каталога
- резервирование баллов при создании заказа
- обработка заказа администратором
- самовывоз как единственный сценарий получения
- in-app уведомления
- базовые email-уведомления, если настроен SMTP
- базовые отчеты и экспорт
- базовый аудит критичных действий

### Что не входит в MVP

- KPI-дашборды и целевые метрики
- начисления руководителями
- workflow согласования начислений и корректировок
- `SSO`, `AD`, `LDAP`, `SAML`, `OIDC`
- интеграции с `HR`, `ERP`, `WMS`, `BI`
- кампании, акции, геймификация, достижения
- рейтинги товаров и обратная связь
- персональные рекомендации
- мультиязычность
- мобильное приложение
- мультиарендность внутри одного развертывания
- продвинутый антифрод
- расширенная аналитика и поиск аномалий

## Зафиксированные продуктовые решения для MVP

### 1. Модель развертывания

`MVP` делаем по модели `one company per deployment`.

Почему так:

- проще разграничение доступа
- проще изоляция данных
- проще брендирование и настройки
- проще внутреннее on-prem внедрение

### 2. Модель учета баланса

Систему не стоит строить вокруг одного изменяемого поля `balance`.

Принятое решение:

- все денежно-подобные операции хранятся в `transaction ledger`
- текущий баланс дополнительно хранится в `balance snapshot`

Это сразу дает базу для:

- истории операций
- резервирования при заказе
- возвратов
- сгорания баллов
- аудита
- будущих сценариев согласования

### 3. Ролевая модель

В `MVP` оставляем только две роли:

- `Employee`
- `Program Admin`

`Program Admin` объединяет обязанности текущего HR-администратора и оператора мерча. Если позже потребуется более тонкое разделение, доменную модель переделывать не придется.

### 4. Модель получения заказа

В `MVP` включаем только `self-pickup`.

Почему так:

- минимальная операционная сложность
- нет адресов доставки
- нет курьерской логистики
- нет интеграций с внешними сервисами

### 5. Модель уведомлений

Базовый канал:

- in-app уведомления

Дополнительно в `MVP`, если есть инфраструктура:

- email-уведомления через SMTP

## Матрица Core / Configurable / Later

| Область | Core в MVP | Настраивается в MVP | Later |
|---|---|---|---|
| Развертывание | Один инстанс на одну компанию | URL, окружения, системные параметры | Полноценный multi-tenant |
| Брендирование | Базовый интерфейс продукта | Название компании, логотип, основной цвет, название валюты | Полный white-label |
| Авторизация | Локальная авторизация | Политика паролей, время сессии | `SSO`, `AD`, `LDAP`, `OIDC` |
| Роли | `Employee`, `Program Admin` | Тонкие переключатели прав админа при необходимости | `Manager`, `Auditor`, `Merch Operator`, `Super Admin` |
| Данные сотрудников | Импорт из Excel, статус активен/неактивен | Шаблон импорта, обязательные колонки | Синхронизация с HR-системой |
| Оргструктура | Департамент как атрибут сотрудника | Филиал, юрлицо, дополнительные поля | Полное дерево оргструктуры и иерархия |
| Баланс | `ledger + snapshot` | Точность, отображаемое имя валюты | Несколько программ мотивации |
| Типы транзакций | Начисление, корректировка, резерв, списание, возврат резерва, сгорание | Справочник причин | Транзакции, связанные с workflow |
| Начисления | Ручное начисление админом | Справочник причин, лимит на операцию, шаблоны комментариев | Массовые начисления, автоначисления |
| Сгорание | Одно глобальное правило | `N` месяцев, срок предупреждения, вкл/выкл механики | Сгорание по типу начисления или кампании |
| Каталог | CRUD товаров, категории, фото, ручные остатки | Категории, теги, скрытые поля | Синхронизация с `ERP/WMS` |
| Заказы | Создание, резерв, подтверждение, сборка, выдача, отмена | Точки выдачи, лимит количества | Доставка, split orders, логистика |
| Уведомления | In-app события | Email-шаблоны, отправитель, включение SMTP | Интеграции с корпоративными коммуникациями |
| Отчетность | Балансы, история транзакций, заказы, экспорт | Фильтры, колонки, формат выгрузки | `BI` API, дашборды, аналитика вовлеченности |
| Аудит | Лог критичных действий | Retention, формат экспорта | Неизменяемый audit trail с diff |
| Безопасность | `RBAC`, валидация, базовое логирование | Политика паролей, лимиты на загрузки | Антифрод, алерты по аномалиям |
| UX | Адаптивный web-интерфейс | Бренд-копирайтинг, ассеты | Персонализация интерфейса |

## Черновик доменной модели

| Сущность | Назначение | Ключевые поля |
|---|---|---|
| `CompanySettings` | Настройки компании в конкретном развертывании | название компании, название валюты, логотип, основной цвет, срок сгорания, дни предупреждения, точки выдачи |
| `User` | Аутентификация и доступ | email или login, password hash, role, status, last login |
| `Employee` | Профиль сотрудника в программе мотивации | табельный номер, ФИО, email, департамент, должность, руководитель, дата приема, статус |
| `BalanceSnapshot` | Быстрое чтение текущего баланса | employee id, available amount, reserved amount, updated at |
| `Transaction` | Неизменяемое событие, влияющее на баланс | type, employee id, amount, status, reason id, order id, comment, author id, effective at |
| `AccrualReason` | Управляемый справочник причин начислений и корректировок | code, title, type, active flag |
| `ProductCategory` | Категория товара | name, active flag, sort order |
| `Product` | Карточка товара | sku, title, description, price in points, stock qty, status, category id |
| `ProductImage` | Фотографии товара | product id, file path, sort order |
| `Order` | Заказ сотрудника | employee id, status, pickup point, total reserved amount, created at, confirmed at, issued at |
| `OrderItem` | Позиции заказа | order id, product id, quantity, price at order time |
| `Notification` | Пользовательское уведомление | user id, type, title, body, read flag, sent at |
| `AuditEvent` | След критичных действий | actor id, action, entity type, entity id, payload, created at |
| `ImportJob` | След выполнения импорта сотрудников | file name, status, rows total, rows succeeded, rows failed, initiated by |

## Обязательные бизнес-правила MVP

1. Любое изменение баланса должно создавать запись в `Transaction`.
2. При создании заказа баллы не списываются окончательно, а резервируются.
3. При подтверждении заказа резерв конвертируется в финальное списание.
4. При отмене или отклонении заказа резерв возвращается через отдельную транзакцию.
5. Ручная корректировка невозможна без причины и комментария.
6. Остаток товара уменьшается только после подтверждения заказа.
7. Сгорание баллов выполняется по расписанию фоновым процессом.
8. Импорт сотрудников валидирует уникальность табельного номера и email.
9. Неактивный сотрудник не может войти в систему и оформить заказ.
10. Действия администратора, меняющие баланс, каталог или статус заказа, обязательно аудируются.

## Основные пользовательские сценарии MVP

### 1. Подключение сотрудника

1. Администратор импортирует сотрудников из Excel.
2. Система валидирует строки и создает записи сотрудников.
3. Для сотрудника создается или привязывается локальная учетная запись.
4. Сотрудник входит в систему и видит баланс, каталог и историю заказов.

### 2. Ручное начисление

1. Администратор выбирает сотрудника.
2. Администратор указывает причину начисления и сумму.
3. Администратор добавляет обязательный комментарий.
4. Система создает транзакцию и обновляет снимок баланса.
5. Сотрудник получает уведомление.

### 3. Ручная корректировка

1. Администратор открывает действия по балансу сотрудника.
2. Администратор вводит положительную или отрицательную корректировку.
3. Система требует причину и комментарий.
4. Система создает транзакцию корректировки и audit event.

### 4. Заказ из каталога

1. Сотрудник открывает каталог.
2. Сотрудник добавляет товар в корзину.
3. Система проверяет доступный баланс и остатки.
4. Сотрудник подтверждает заказ и выбирает точку самовывоза.
5. Система создает заказ и транзакцию резерва.

### 5. Обработка заказа

1. Администратор открывает новые заказы.
2. Администратор подтверждает наличие товара.
3. Система переводит резерв в финальное списание.
4. Администратор двигает заказ по статусам до `Issued`.
5. Сотрудник получает уведомления об изменении статуса.

### 6. Отмена заказа

1. Администратор или система отменяет заказ до выдачи.
2. Система создает транзакцию возврата резерва.
3. Зарезервированная сумма возвращается в доступный баланс.
4. Заказ получает конечный статус `Canceled`.

### 7. Сгорание баллов

1. Планировщик находит начисления, подходящие к сроку сгорания.
2. Система создает транзакции сгорания.
3. Система обновляет снимок баланса.
4. Сотрудник заранее получает предупреждение по настройкам компании.

## Решения, которые стоит закрепить на следующей сессии

Эти пункты не блокируют текущее продуктовое рамочное решение, но нужны до архитектуры и оценки сроков.

1. Какой вариант локальной авторизации используем в `MVP`: инвайт, пароль от админа или восстановление через email.
2. Нужно ли уже в `MVP` делить `Program Admin` на набор отдельных permission toggles в интерфейсе.
3. Остатки обязательны для всех товаров или допускаются позиции с бесконечным остатком.
4. Разрешаем ли сотруднику самостоятельно отменять заказ до подтверждения администратором.
5. Какой Excel-шаблон импорта сотрудников станет стандартом продукта.

## Критерий завершения текущего этапа

Этап можно считать завершенным, когда команда согласовала:

- границы `MVP`
- продуктовые решения выше
- матрицу `Core / Configurable / Later`
- черновик доменной модели
- порядок эпиков для реализации

Следующий логичный шаг после этого документа: архитектурное проектирование и техническая декомпозиция.
